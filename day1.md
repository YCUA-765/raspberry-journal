## 🔧 硬件组装

### 散热片

树莓派5发热量大，**通电前必须先贴散热片**。

| 散热片 | 对应芯片 |
|--------|----------|
| 最大片 | Broadcom 主CPU（中间银色大方块）|
| 第二大片 | 内存芯片（旁边黑色方块）|
| 小片 × 2 | 左下 / 右下小芯片 |

背面自带导热双面胶，撕开直接贴，按压几秒固定。

### TF卡

插在主板背面，金属触点朝向主板，推到底听到卡扣声即可。出厂已预装镜像，无需自己烧录。

### 接线原则

```
先接所有外设线
最后才接电源    ← 不要带电插拔任何配件
```

### 亚克力外壳 + 风扇

网络问题暂时搁置后，趁机把外壳和风扇装完：

- **外壳**：透明亚克力，主板固定后各接口对准开口
- **风扇**：4针接口插主板风扇座（靠近USB口），螺丝固定在外壳顶部，通电后按CPU温度自动调速

## 🔴 踩坑实录：远程连接失败

### 初始方案

卖家提供系统信息：热点 `Pi_Hot`，树莓派IP `10.42.0.1`。

```bash
ssh pi@10.42.0.1
# → ssh: connect to host 10.42.0.1: Network is unreachable
```

### 排查过程

```bash
networksetup -listallhardwareports
# WiFi 对应物理接口：en0（Mac内置WiFi天线）

ifconfig en0 | grep inet
# inet 192.168.1.22  ← 不对，应该是 10.42.0.x

route get 10.42.0.1
# gateway: 192.168.1.11  ← 路由表把流量送到了校园网网关
```

### 根因分析

问题出在三个层面同时叠加：

**第一层：接口冲突**

Mac 的 WiFi 物理接口 `en0` 被校园网和 Pi_Hot 同时共用。每个接口有独立的IP地址，`en0` 同时服务两个网络，产生冲突。

**第二层：DHCP 被抢占**

校园网 DHCP 先给 `en0` 分配了 `192.168.1.22`，Pi_Hot 的 DHCP 后来发出分配请求时，Mac 认为 `en0` 已有IP，直接忽略——导致 Mac 在 Pi_Hot 网段（`10.42.0.x`）没有合法IP。

**第三层：路由表走错门**

```
目标：10.42.0.1
查路由表：无精确规则
→ 命中默认路由
→ 发给校园网网关 192.168.1.11
→ 网关不认识 10.42.0.1
→ Network unreachable
```

### 尝试了什么

| 方案 | 结果 | 原因 |
|------|------|------|
| `sudo route add 10.42.0.0/24 -interface en0` | `Host is down` | 路由方向对了，但源IP仍是 `192.168.1.22`，树莓派回包找不到路，丢包 |
| `vnc://10.42.0.1` | 连接失败 | 同样的网络问题，换协议没有用 |
| bootfs 写入 `wpa_supplicant.conf` | 未生效 | 定制镜像不识别该配置文件 |

> 💡 `unreachable` = 包没出去；`host is down` = 包出去了但没有回应。两者是不同的失败层级。

## 📋 明天的计划

购买 **USB-C Hub（含网口 + HDMI）** + **7寸小显示屏**，改用网线直连：

## 💭 今日收获

**技术层面**

| 概念 | 理解 |
|------|------|
| 网络接口 | Mac 与网络之间的连接点，每个接口独立工作，有独立IP |
| DHCP | 自动分配IP的服务，先到先得，后来的请求会被忽略 |
| 路由表 | 决定数据包从哪个接口出去的规则表，从精确到模糊匹配 |
| 网段 | IP地址的网络部分，同网段设备可直接通信，跨网段需要网关转发 |

**认知层面**

第一次真实遇到「硬件 + 网络 + 软件」三层同时出问题的感觉。不是代码报错，而是整个环境出了问题，需要逐层隔离排查。

> 工科思维原则：**能用硬件解决的问题，不要用软件绕。** 今天花了大量时间用软件方案解决硬件资源不足的问题。明天买到硬件，10分钟解决。
